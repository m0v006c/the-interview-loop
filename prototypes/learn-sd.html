Architecture Overview

Before we build, let's sketch the full picture. A notification system has three jobs: accept events from anywhere, route them to the right channel, and deliver reliably even when things break.

Concepts introduced in this section
Event-driven architecture Producer / Consumer Idempotency Fan-out pattern

The five core components

Our system needs five layers. Each one has a single responsibility, which makes the system testable, scalable, and replaceable.

1
Event Ingestion API
Accepts events from any upstream service. REST or gRPC. Must be fast and non-blocking — it should acknowledge immediately and hand off to the queue.
2
Message Queue (Kafka)
Decouples ingestion from delivery. Consumers read at their own pace. Retained messages enable replay on failure — we'll deep-dive this in Section 3.
3
Notification Router
Reads user preferences and routes each event to the correct channels: push, email, SMS, in-app. This is where the fan-out pattern lives.
4
Channel Delivery Workers
One worker per channel type. Each wraps a third-party provider (FCM, SendGrid, Twilio). Circuit breakers sit here — covered in Section 5.
5
Notification Store
PostgreSQL for user preferences and delivery logs. Redis for deduplication and delivery status. Supports the inbox / "mark as read" UX.

Database schema

Two critical tables. notification_events is append-only — never update, only insert. This sets us up for event sourcing later. 

-- Append-only event log (never UPDATE this table)
CREATE TABLE notification_events (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      BIGINT      NOT NULL,
  event_type   VARCHAR(64) NOT NULL,  -- e.g. 'order.shipped', 'friend.request'
  payload      JSONB       NOT NULL,
  idempotency_key VARCHAR(128) UNIQUE, -- prevents duplicate sends
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- User preferences (which channels for which event type)
CREATE TABLE notification_preferences (
  user_id      BIGINT NOT NULL,
  event_type   VARCHAR(64) NOT NULL,
  channel      VARCHAR(32) NOT NULL,  -- 'push', 'email', 'sms', 'in_app'
  enabled      BOOLEAN NOT NULL DEFAULT true,
  PRIMARY KEY (user_id, event_type, channel)
);

-- Delivery log (one row per attempt per channel)
CREATE TABLE delivery_log (
  id           BIGSERIAL PRIMARY KEY,
  event_id     UUID REFERENCES notification_events(id),
  channel      VARCHAR(32) NOT NULL,
  status       VARCHAR(16) NOT NULL,  -- 'pending', 'sent', 'failed', 'bounced'
  provider_id  VARCHAR(128),          -- FCM message ID, SendGrid ID, etc.
  attempted_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  error_msg    TEXT
);
💡 Why the idempotency key?

Networks fail. Retry logic is essential. But if we retry without an idempotency key, a user gets the same push notification three times. The UNIQUE constraint on idempotency_key guarantees the database rejects duplicates even if your application layer retries. The caller generates this key — typically SHA256(user_id + event_type + timestamp_bucket).

The ingestion API — code walkthrough

The API accepts events, validates, persists, and immediately enqueues. It must return in under 10ms — so no synchronous delivery here. 

@RestController
@RequestMapping("/v1/notifications")
public class NotificationIngestionController {

    private final NotificationEventRepository eventRepo;
    private final KafkaTemplate<String, NotificationEvent> kafka;

    @PostMapping("/events")
    public ResponseEntity<NotificationResponse> ingestEvent(
            @RequestBody @Valid NotificationRequest req) {

        // 1. Persist first — Kafka publish is NOT transactional with the DB
        NotificationEvent event = NotificationEvent.builder()
            .userId(req.getUserId())
            .eventType(req.getEventType())
            .payload(req.getPayload())
            .idempotencyKey(req.getIdempotencyKey())
            .build();

        try {
            eventRepo.save(event);  // throws DataIntegrityViolationException on duplicate
        } catch (DataIntegrityViolationException e) {
            // Duplicate — return 200 as if we processed it (idempotent response)
            return ResponseEntity.ok(NotificationResponse.alreadyProcessed(req.getIdempotencyKey()));
        }

        // 2. Publish to Kafka after successful DB write
        kafka.send("notification-events", event.getUserId().toString(), event);

        return ResponseEntity.accepted().body(NotificationResponse.accepted(event.getId()));
    }
}
Notice: We persist to the database before publishing to Kafka. If Kafka is unavailable, the event is still saved. A separate outbox processor can retry the Kafka publish later. This is the Transactional Outbox pattern — we'll cover it fully in Section 7 (Event Sourcing).
🎯 Interview tip — this section

Interviewers test whether you know the difference between fire-and-forget and reliable delivery. When asked "how do you ensure every notification is delivered?", walk through: persist first → enqueue → worker picks up → retry on failure → dead-letter queue for unrecoverable failures. That answer alone demonstrates senior-level thinking.
